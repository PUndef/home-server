# AdGuard Home: пошаговый план для home-server

> **Статус (2026-04): снято с эксплуатации.** AdGuard Home не устраивали неполная блокировка и тормоза; контейнер LXC **102** **удалён в Proxmox**. Ниже — **архивный** план и история настроек; на текущую схему сети не опираться как на «активную».

Цель (исторически): включить DNS-фильтрацию рекламы через `AdGuard Home` в домашней сети, чтобы на `Wi‑Fi` все клиенты автоматически использовали DNS.

Дополнительно: отдельно описано, как сделать то же для `мобильного интернета` (LTE/5G) — но там есть развилки, потому что это требует либо VPN, либо публичного DoH/endpoint.

## Часть A. Поднимаем `AdGuard Home` локально (только LAN / Wi‑Fi)

### Шаг A1. Создай хост в Proxmox под AdGuard (VM или LXC)
**Фактически у тебя:** LXC Debian `102` (helper-script `bash -c "$(curl .../ct/debian.sh)"` из [Proxmox VE Helper Scripts](https://community-scripts.org/)).

Альтернатива — отдельная VM (например `vm/debian-vm.sh` из того же проекта или ручной образ).

1. Ресурсы (минимум): `1 vCPU`, `512MB–1GB RAM`, диск `8–12GB`.
2. Сеть: `vmbr0`, статический или DHCP с резервацией; итоговый IP AdGuard: **`192.168.50.225`**.

### Шаг A2. Проверь связность в LAN
С любого устройства в твоей `192.168.50.0/24`:
1. Убедись, что VM пингуется по IP.
2. Проверь, что web/панель ещё не настроена (пока это нормально).

### Шаг A3. Установи AdGuard Home (предпочтительно через Docker)
В твоём случае AdGuard уже находится/нужен в LXC-контейнере `102` (IP `192.168.50.225`), поэтому проще всего поставить его “как есть” скриптом от AdGuardTeam.

#### A3.1. Установка через официальный install.sh
1. Зайди внутрь контейнера `102` и выполни под `root` (или через `sudo -i`).
2. Запусти:
```sh
curl -s -S -L https://raw.githubusercontent.com/AdguardTeam/AdGuardHome/master/scripts/install.sh | sh -s -- -v
```

#### A3.2. После установки: проверь, что сервис поднялся
Обычно достаточно:
- открыть панель: `http://192.168.50.225:3000`
- проверить статус сервиса командой `systemctl status ...` (точное имя unit может отличаться)

Если увидишь ошибки вида `bind :53` / `permission denied`, это частая ситуация для `unprivileged LXC` — тогда AdGuard DNS на `53` не поднимется. В этом случае лучше перенести AdGuard в VM (там эти ограничения не такие строгие).

### Шаг A4. Открой веб-панель AdGuard Home локально
С телефона/ПК в Wi‑Fi:
1. Сразу после `install.sh` часто доступно: `http://192.168.50.225:3000`. После мастера, если в мастере выбран порт **80** для админки: `http://192.168.50.225` (без порта).
2. Пройди мастер настройки:
   - задай логин/пароль админки
   - в upstream DNS впиши обычные резолверы на старте (например `1.1.1.1`, `8.8.8.8`)
   - включи рекомендованные фильтры
3. Включи логирование запросов (или просто проверь, что в `Logs/Query log` идут записи).

#### A4.1. Убедись, что AdGuard слушает LAN-интерфейс
В настройках `DNS settings` -> `Interfaces/Listeners` проверь, что слушатель включает либо `0.0.0.0`, либо именно IP VM `192.168.50.225`.

### Шаг A5. Проверка “работает ли блокировка” без роутера
Чтобы убедиться, что AdGuard реально резолвит:
1. На Android подключись к Wi‑Fi.
2. Временно настрой DNS вручную на Wi‑Fi (если есть пункт):
  - DNS server: `192.168.50.225`
3. Открой сайт/сервис, где обычно видно рекламу/трекинг.
4. Вернись в AdGuard: убедись, что запросы видны.

Если AdGuard в логах видит DNS-запросы — часть A готова.

## Часть B. Настраиваем ASUS RT‑AX55, чтобы “автоматически на всём Wi‑Fi”

### Шаг B1. Найди DHCP DNS настройку в ASUS RT‑AX55
В админке роутера:
1. `LAN` -> `DHCP Server`
2. Проверь что DHCP сервер включен.
3. Найди поле типа:
   - `DNS Server`
   - или `Option 6 (DNS)`
   - или “DHCP Option: DNS”

### Шаг B2. Укажи DNS как IP AdGuard VM
1. Впиши `192.168.50.225` (твои IP AdGuard)
2. Сохрани настройки.
3. Для надёжности:
   - отключи/включи Wi‑Fi на телефоне
   - или “выпишися/впишися” в сеть Wi‑Fi
   - или перезагрузи телефон

### Шаг B3. Android-нюанс “Private DNS”
Чтобы Android НЕ обходил DHCP DNS:
1. `Настройки` -> `Сеть и интернет` -> `Частный DNS`
2. Поставь `Выкл.` или отключи пользовательский Private DNS provider

После этого телефон должен начинать использовать DNS из DHCP (то есть AdGuard).

### Шаг B4. Финальная проверка
1. Открой в AdGuard `Query log`.
2. Открой браузер на телефоне.
3. Убедись, что идут запросы и резолв происходит через AdGuard.

## Часть C. Мобильный интернет (LTE/5G): как сделать, чтобы тоже блокировало

Без WireGuard “автоматически” на LTE обычно не получится, потому что DHCP от домашнего роутера на LTE недоступен.

Дальше 2 легальных пути:

### Вариант C1 (рекомендовано): VPN (чтобы весь DNS/трафик шёл через дом)
Ставишь VPN на Android (например WireGuard) до дома, и клиенты внутри VPN начинают пользоваться домашним DNS/AdGuard.

Плюсы: стабильно, не надо публично открывать DNS.
Минусы: нужна настройка VPN.

> Если хочешь — сделаю отдельный файл “WireGuard + AdGuard через VPN” под твою схему.

### Вариант C2: публичный DoH/DoT endpoint от AdGuard (без VPN)
Идея: AdGuard должен быть доступен извне по HTTPS (DoH), а Android должен быть настроен на “Private DNS” через hostname.

Что нужно:
1. Домен (например `dns.<твой_домен>`)
2. TLS сертификат для HTTPS endpoint
3. Проброс/реврайт на роутере и/или reverse proxy на том же `443` (у тебя `443` уже занят Nextcloud, поэтому придётся аккуратно сделать поддомен/маршрут)

Что проверить заранее:
1. Есть ли у тебя возможность добавить ещё один virtual host на `443` для поддомена
2. Где сейчас обслуживается `443` (на `nextcloud-vm` или на роутере)

Без этих деталей я не буду “тыкать” команды и пробросы вслепую (чтобы не сломать Nextcloud).

## Дальше (когда понадобится)

- **Только LAN:** при желании включи DHCP DNS на ASUS (часть B) и ограничь доступ клиентов `192.168.50.0/24`.
- **LTE / из интернета:** см. часть C — WireGuard домой или публичный DoH с защитой (не открывать «голый» 53 в интернет).

Замер upstream с хоста AdGuard (одной строкой в shell на Debian):

```bash
candidates=("77.88.8.1" "77.88.8.8" "1.1.1.1" "1.0.0.1" "8.8.8.8" "8.8.4.4" "9.9.9.9" "149.112.112.112"); domain="example.com"; runs=10; for dns in "${candidates[@]}"; do sum=0; ok=0; fail=0; for i in $(seq 1 $runs); do ms=$(dig @"$dns" "$domain" +tries=1 +timeout=2 +stats 2>/dev/null | awk '/Query time:/ {print $4}'); if [ -n "$ms" ]; then sum=$((sum+ms)); ok=$((ok+1)); else fail=$((fail+1)); fi; done; if [ "$ok" -gt 0 ]; then avg=$((sum/ok)); echo "$avg ms avg | ok=$ok fail=$fail | $dns"; else echo "9999 ms avg | ok=0 fail=$fail | $dns"; fi; done | sort -n
```

Перед первым запуском: `apt update && apt install -y dnsutils`.
